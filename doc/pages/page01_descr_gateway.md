## Загальні принципи {#sec_descr_gateway_common}

Сервер `modbusua` реалізує проміжний рівень між існуючими промисловими контролерами/ПЛК,
які підтримують протокол Modbus, та SCADA-системами або іншими програмами верхнього рівня,
які підтримують протокол OPC UA.

Основною метою сервера `modbusua` є збір технологічних даних із підключених пристроїв
через протокол Modbus, їх обробка та подальша передача через універсальний протокол OPC UA.
Сервер `modbusua` реалізовано як кросплатформенну (Windows, Linux) консольну програму/сервіс
(демон у термінології Unix) з відкритим вихідним кодом на C++.

Сервер `modbusua` підтримує роботу з різними типами пристроїв/ПЛК, які використовують
протокол Modbus TCP/RTU/ASCII, та забезпечує надійний і ефективний обмін даними 
(читання/запис) між цими пристроями та клієнтами OPC UA.

Налаштування параметрів з’єднання, таймаутів і повторних спроб здійснюється централізовано
через конфігураційний файл: для кожного порту та/або пристрою визначаються адреса/хост, номер 
порту й тип транспорту (TCP/RTU/ASCII), окремі таймаути на встановлення з’єднання та на 
операції читання/запису, а також обмеження черги; політика повторів включає кількість спроб, 
інтервал між ними, загальний таймаут транзакції та пороги відключення/переініціалізації.

Сервер `modbusua` підтримує гаряче оновлення конфігурації без зупинки роботи: зміни з файлу 
`modbusua.conf` можуть бути застосовані через OPC UA команду `ReloadConfig` або надсилання 
сигналу `SIGHUP` (через `systemd` параметр `ExecReload`); нові параметри проходять валідацію і 
атомарно вводяться в дію по областях (глобальні налаштування, порти, пристрої) без розриву 
чинних TCP/IP з’єднань; цикл обміну коротко синхронізується лише для тих компонентів, яких 
стосуються зміни, дані та буфери стану зберігаються, а статус і результати операції детально 
журналюються; у разі помилок валідації застосування відхиляється, сервер продовжує роботу з 
попередньою конфігурацією.

Вбудовані механізми діагностики та моніторингу стану сервера і з’єднань забезпечують 
видимість у реальному часі: доступні детальне журналювання з рівнями та ротацією, 
лічильники успішних/неуспішних операцій, час останнього обміну, статус з’єднання, 
середній/максимальний час відповіді й коди помилок; ці показники відображаються 
через OPC UA вузли `Stat` на рівні сервера, портів і пристроїв, передбачено команди 
`StatClear` для очищення лічильників; додатково підтримується вивід у системні журнали ОС 
(console/syslog) і структуровані логи для аналітики, що спрощує діагностику інцидентів та 
проактивне виявлення деградацій каналу.

Підтримка різних типів даних реалізована на рівні перетворення Modbus-реєстрів у стандартні
OPC UA значення: сервер коректно інтерпретує дискрети/біти, цілі числа зі знаком і без
(8/16/32/64 біт), числа з плаваючою комою (IEEE‑754 32/64), враховує порядок слів (word swap) 
та об’єднання регістрів у багатослівні типи.

## Загальний опис сервера вводу-виводу {#sec_descr_gateway_server}

Розробка драйвера передбачає створення проміжного сервера, який буде зчитувати дані із існуючих контролерів
з використанням Modbus TCP, RTU або ASCII, а потім віддавати ці дані
з використанням існуючих внутрішніх механізмів обміну між стандартними вузлами SCADA,
в даному випадку OPC UA клієнтом, який входить у пакет драйверів будь-якої сучасної SCADA системи.

Даний сервер передбачає можливість створення обміну з заданою кількістю існуючих контролерів і є конфігурованим.
Конфігурація задається за допомогою файлу конфігурації `modbusua.conf`.

Конфігурація складається з двох основних об’єктів: об’єкту порту та об’єкту пристрою/ПЛК.
Об’єкт порту включає в себе мережеві налаштування процесу обміну, такі як адреса пристрою в мережі, номер порту тощо.
Об’єкт-порт відповідає одному TCP/IP з’єднанню. Об’єкт пристрій включає в себе налаштування конкретного пристрою/ПЛК.
Таких об’єктів пристроїв для одного порту може бути безліч, і відповідно обмін із кожним пристроєм буде виконуватись
послідовно. Таким чином використувуючи один порт та декілька пристроїв можна будувати стандартну Modbus-мережу.

В процесі роботи драйвера передбачено використання журналу роботи даного сервера вводу-виводу.
Це означає, що всі дії сервера по обміну із існуючими контролерами, по обміну із клієнтами що підключаються до нього
і т.д. записуються у стандартний журнал ОС, виводяться у консоль або у файл для контролю за роботою даного сервера,
а також для виявлення помилок конфігурації, підключення та ін.
Рівень журналювання (тобто кількість інформаційних повідомлень, що виводиться) задається через та способи виводу 
встановлюються у тому самому файлі конфігурації сервера `modbusua.conf`.

## Реалізація клієнтського протоколу Modbus {#sec_descr_gateway_protocol}

Протокол Modbus є стандартним промисловим протоколом для обміну даними між пристроями автоматизації.
Сервер `modbusua` реалізує клієнтську частину протоколу Modbus, що дозволяє йому взаємодіяти з різними
контролерами/ПЛК, які підтримують цей протокол. Сервер підтримує різні варіанти транспорту Modbus,
включаючи Modbus TCP, Modbus RTU та Modbus ASCII, що забезпечує гнучкість у підключенні до різних типів пристроїв.

Клієнтська реалізація Modbus у сервері `modbusua` включає підтримку зчитування всіх типів пам’яті Modbus:
- `FC01. READ_COILS` - зчитування значень котушок (Coils, `0x`-пам'ять, дискретні виходи)
- `FC02. READ_DISCRETE_INPUTS` - зчитування значень дискретних входів (Discrete Inputs, `1x`-пам'ять)
- `FC03. READ_HOLDING_REGISTERS` - зчитування значень вихідних регістри (Holding Registers, `4x`-пам'ять)
- `FC04. READ_INPUT_REGISTERS` - зчитування значень вхідних регістри (Input Registers, `3x`-пам'ять).

Для запису даних підтримуються наступні функції:
- `FC15. WRITE_MULTIPLE_COILS` - запис множинних котушок (Coils)
- `FC16. WRITE_MULTIPLE_REGISTERS` - запис множинних регістрів (Holding Registers).

Найнижчий пріорітет надається Modbus-посилкам прямого читання даних (poll).
Такі Modbus-посилки ставляться в чергу і оброблюються в паузах якщо Modbus посилки 
для запису у Modbus регістри відсутні. При цьому якщо фіксується обрив зв’язку з ПЛК для всіх посилок
у черзі виставляється статус помилки і дана черга очищується.
В іншому випадку такі посилки будуть знаходитись у черзі доки не з’явиться можливість їх обробки.

Також слід зазначити, що Modbus-посилки прямого запису даних (poke) мають вищий пріорітет
ніж Modbus посилки читання даних контролера або Modbus пристрою.

## Реалізація серверного протоколу OPC UA {#sec_descr_opcua_protocol}

OPC UA (Open Platform Communications Unified Architecture) є сучасним промисловим стандартом для
безпечного та надійного обміну даними між системами автоматизації. На відміну від попередніх версій OPC,
які базувались на технологіях Microsoft COM/DCOM, OPC UA є платформно-незалежним протоколом, що підтримує
різні транспортні механізми та забезпечує вбудовані засоби безпеки. Протокол використовує клієнт-серверну 
архітектуру, де сервер надає структурований адресний простір з ієрархією об'єктів, змінних та методів,
а клієнти можуть зчитувати дані, підписуватися на зміни та викликати методи на сервері.

У сервері `modbusua` протокол OPC UA реалізовано як основний інтерфейс для передачі технологічних даних від
контролерів та ін. Modbus пристроїв (наприклад різні промислові датчики та лічильники) до SCADA систем та
інших промислових додатків. Сервер організовує дані,отримані через Modbus протоколи,
у стандартизований адресний простір OPC UA. Структура адресного простору включає окремі гілки 
для різних типів подій: загальні сигнали, стани засувок, аналогові виміри та системні повідомлення.
Клієнти OPC UA можуть підписуватися на зміни конкретних вузлів та отримувати сповіщення
в реальному часі при надходженні нових подій від контролерів, що забезпечує ефективну та
своєчасну передачу технологічної інформації.

Структура адресного простору сервера OPC UA в `modbusua`:

```
Root
\-- Objects
    +-- Server
    \-- modbusua
        +-- Cfg
        |   +-- LogConsoleFormat
        |   +-- LogConsoleTimeformat
        |   \-- ...
        +-- Cmd
        |   \-- ReloadConfig
        \-- Ports
            +-- Port_1
            |   +-- Cfg
            |   |   +-- ...
            |   |   +-- Host
            |   |   +-- Port
            |   |   \-- ...
            |   +-- Cmd
            |   |   \-- StatClear
            |   +-- Set
            |   |   \-- Enable
            |   +-- Stat
            |   |   +-- ...
            |   |   +-- BadCount
            |   |   +-- GoodCount
            |   |   \-- ...
            |   \-- Devices
            |       |-- Device_1
            |       |   +-- Events
            |       |   +-- Cfg
            |       |   |   +-- ...
            |       |   |   +-- DefaultPeriod
            |       |   |   \-- ...
            |       |   +-- Cmd
            |       |   |   +-- StatClear
            |       |   |   \-- ...
            |       |   +-- Set
            |       |   |   \-- Enable
            |       |   +-- Stat
            |       |   |   +-- ...
            |       |   |   +-- BadCount
            |       |   |   +-- GoodCount
            |       |   |   \-- ...
            |       |   |-- Items
            |       |   |   +-- ...
            |       |   |   \-- ...
            |       |   \-- Messages
            |       |       +-- ...
            |       ...     \-- ...
            \-- Port_2
                |
                ...

```

Структура адресного простору сервера OPC UA в `modbusua` організована у вигляді ієрархічного дерева,
яке забезпечує логічну організацію всіх технологічних даних та системних параметрів.
Адресний простір побудовано згідно зі стандартами OPC UA та включає як стандартні системні об'єкти,
так і специфічні для `modbusua` вузли.

### Кореневий рівень (Root)

Стандартний OPC UA адресний простір починається з кореневого вузла `Root`, який містить
стандартний об'єкт `Objects` - основний контейнер для всіх об'єктів додатку в OPC UA сервері.

### Системний рівень (Objects/Server)

Вузол `Server` містить стандартну системну інформацію OPC UA сервера, включаючи статус сервера,
час запуску, поточний час та рівень сервісу. Ці дані використовуються клієнтами для
моніторингу стану сервера.

### Основний об'єкт додатку (Objects/modbusua)

Головний об'єкт `modbusua` є контейнером для всієї функціональності додатку і включає три основні розділи:

- **Cfg (Configuration)** - містить параметри конфігурації всього сервера, такі як
формат виводу в консоль (`LogConsoleFormat`), формат часу для журналювання (`LogConsoleTimeformat`)
та інші глобальні налаштування системи.

- **Cmd (Commands)** - містить системні команди, зокрема `ReloadConfig` для перезавантаження
конфігурації сервера без зупинки роботи.

- **Ports** - є контейнером для всіх портів комунікації, через які здійснюється зв'язок
з промисловими контролерами та ін. пристроями, що підримують протокол Modbus.

### Рівень портів (modbusua/Ports/Port_N)

Кожен порт представляє собою окрему комунікаційну лінію (TCP/IP з'єднання) і
містить чотири основні розділи:

- **Cfg** - параметри конфігурації порту, включаючи мережеву адресу хоста (`Host`),
номер порту (`Port`), таймаути та інші налаштування з'єднання
(див. [Конфігурація порту](#sec_port_cfg)).

- **Cmd** - команди управління портом, зокрема `StatClear` для очищення статистики роботи порту
(див. [Команди порту](#sec_port_cmd)).

- **Set** - налаштування поточного стану порту, включаючи параметр `Enable` для
увімкнення/вимкнення роботи порту
(див. [Налаштування порту](#sec_port_set)).

- **Stat** - статистична інформація про роботу порту, включаючи лічильники успішних
(`GoodCount`) та неуспішних (`BadCount`) операцій обміну.
(див. [Статистика/діагностика порту](#sec_port_stat)).

### Рівень пристроїв (Ports/Port_N/Devices/Device_N)

Кожен пристрій представляє собою окремий промисловий контролер або ПЛК, підключений
через відповідний порт. Структура пристрою аналогічна структурі порту і включає:

- **Events** - серіалізований рядок, що містить список подій, що має бути вичитаний і
потім онулений клієнтом, щоб сервер міг сформувати наступний рядок подій
(див. [Буфер повідомлень пристрою/ПЛК](#sec_device_ev)).

- **Cfg** - конфігурація пристрою з параметрами типу період опитування за замовчуванням
(`DefaultPeriod`), адреси Modbus, розміри буферів тощо.
(див. [Конфігурація пристрою/ПЛК](#sec_device_cfg)).

- **Cmd** - команди управління пристроєм, включаючи ініціалізацію (`Init`), очищення статистики
(`StatClear`), синхронізацію бази даних, встановлення часу та інші системні операції.
(див. [Команди пристрою/ПЛК](#sec_device_cmd)).

- **Set** - поточні налаштування пристрою, зокрема `Enable` для увімкнення/вимкнення обміну
з даним пристроєм.
(див. [Налаштування пристрою/ПЛК](#sec_device_set)).

- **Stat** - статистика роботи пристрою з лічильниками успішних та неуспішних операцій,
часом останнього обміну, статусом з'єднання тощо.
(див. [Статистика/діагностика пристрою/ПЛК](#sec_device_stat)).

- **Items** - містить прямі Modbus дані (дискретні входи/виходи, аналогові входи/виходи),
які зчитуються циклічно згідно з конфігурацією.
(див. [Modbus регістри](#sec_device_mb)).

- **Messages** - містить структуру внутрішніх повідомлень Modbus. Дозволяє діагностувати всі
внутрішні Modbus посилки, що формуються для роботи з Modbus даними прямого читання/запису.

Така ієрархічна структура забезпечує логічну організацію даних від загальносистемного рівня
до конкретних технологічних параметрів, дозволяє легко навігувати адресним простором та
надає клієнтам OPC UA зручний доступ до всієї необхідної інформації для моніторингу та
управління промисловими процесами.

## Опції командного рядка {#sec_descr_gateway_cmdoption}

Доступні наступні опції командного рядку для виконуваної програми `modbusua`:

| Опція             | Скор. | Додат. парам. | Опис
|:------------------|:------|:--------------|-------------------------------------------
| `--version`       | `-v`  | `-`           | версія програми
| `--help`          | `-?`  | `-`           | допомога
| `--file`          | `-f`  | `<filename>`  | задати файл конфігурації (за замовч. `modbusua.conf`)

## Встановлення на ОС Windows {#sec_descr_gateway_win}

Для встановлення `modbusua` використовується `modbusua_<version>_<arch>.msi` інсталятор.
Інсталятор створює всі необхідні файли у папці встановлення (за замовчуванням `C:\Program Files\modbusua`),
встановлює `modbusua` у список програм Windows а також реєструє `modbusua` як сервіс Windows.

Додатково, для встановлення/видалення `modbusua` із списку сервісів Windows може застосовуватись
допоміжна утиліта `modbusuaservice.bat`, яка необхідно запустити із командного рядка
у режимі адміністратора і використати опції командного рядку `-i`/`-u` відповідно
(`-n` задає альтернативне ім'я для сервісу, `-?` або `--help` для показу допомоги).

Наприклад:
```console
> modbusuaservice.bat -i -n My_modbusua
```

## Встановлення на ОС Linux {#sec_descr_gateway_linux}

Для встановлення `modbusua` на ОС Linux використовується пакетний інсталятор `modbusua_<version>_<arch>.deb`
для дистрибутивів на основі Debian/Ubuntu (для пакетного менеджера `dpkg`), або
`modbusua-<version>.<arch>.rpm` для дистрибутивів на основі RedHat/Fedora/CentOS.

Файл `modbusua.service` визначає параметри для роботи драйвера як сервісу у ОС Linux.
Він призначений для використання у менеджері сервісів `systemd`.
Цей файл має бути розміщений у папці `/etc/systemd/system`.

Основні параметри, що задаються у файлі `modbusua.service`:
* _`Type`_ - повинен мати тип `forking`, так як драйвер створює дочірній процес як основний робочий процес,
а процес який `systemd` запускає (батьківський) автоматично завершується;
(через станд. Unix функцію `daemon()`);
* _`ExecStart`_ - задає файл для запуску програми драйвера та його параметри.
За замовчуванням це `/usr/bin/modbusua -f /etc/modbusua/modbusua.conf`.
Відповідно `/usr/bin/modbusua` розташування виконуваного файлу програми,
`-f /etc/modbusua/modbusua.conf` - параметри командного рядка, що вказуються файл конфігурації;
* _`ExecReload`_ - команда яку `systemd` повинен виконати,
щоб перезавантажити файл конфігурації `modbusua.conf`.
За замовчуванням це `kill -HUP $MAINPID`;
* _`WorkingDirectory`_ - робоча директорія програми, відносно якої визначаються 
відносні шляхи, який присутні у конфігурації.
За замовчуванням це `/var/lib/modbusua`.

Приклад вмісту файлу `modbusua.service`:
```conf
[Unit]
Description=OPC UA - Modbus Gateway
After=network.target

[Service]
Type=forking
ExecStart=/usr/bin/modbusua -f /etc/modbusua/modbusua.conf
ExecReload=kill -HUP $MAINPID
WorkingDirectory=/var/lib/modbusua

[Install]
WantedBy=multi-user.target
```

Для роботи програми як сервісу необхідно щоб всі файли знаходились на своїх місцях
і щоб всі необхідні директорії (напр. `WorkingDirectory`) були створені.

### Основні команди керування modbusua-сервісом в ОС Linux {#sec_descr_gateway_linux_cmd}

Далі представлені команди консолі для керування modbusua-сервісом:

* перегляд всіх доступних сервісів:
```console
$ systemctl list-units
```

* перегляд статусу modbusua-сервісу:
```console
$ systemctl status modbusua
```

* перегляд unit-файлу сервісу `modbusua.service`:
```console
$ systemctl cat modbusua
$ systemctl --no-pager cat modbusua
```

* запуск modbusua-сервісу:
```console
$ sudo systemctl start modbusua
```

* зупинка modbusua-сервісу:
```console
$ sudo systemctl stop modbusua
```

* видача команди на зміну конфігурації (необхідна після зміни файлу `modbusua.conf`):
```console
$ sudo systemctl reload modbusua
```

* команда на перезапуск працюючого сервісу (необхідна після зміни файлу `modbusua.service`):
```console
$ sudo systemctl daemon-reload modbusua
```

* додати modbusua-сервіс до автозавантаження при старті ОС:
```console
$ sudo systemctl enable modbusua
```

* видалити modbusua-сервіс із автозавантаження при старті ОС:
```console
$ sudo systemctl disable modbusua
```

Загальний вигляд файлів `modbusua` у файловій системі Linux:
```
/                                # Коренева директорія файлової системи Linux
+-- usr/                         #         
|   +-- bin/                     #
|       +-- modbusua             # Файл виконуваної програми сервера вводу-виводу
+-- etc/                         #         
|   +-- modbusua.conf            # Файл конфігурації
|   +-- systemd/                 #
|       +-- system/              #
|           +-- modbusua.service # Файл сервісу для systemd
\-- var/                         #
    +-- lib/                     #
    |   +-- modbusua/            # Робоча директорія сервера вводу-виводу
    +-- log/                     #
        +-- modbusua/            # Папка для збереження лог-файлів
```
